# Web Accessibility (WCAG 2.2 AA)

## Applicability

WCAG 2.2 AA is the global baseline standard for web accessibility. Legal enforcement varies by jurisdiction:

- **EU: European Accessibility Act (EAA)** — enforced from June 28, 2025. Applies to e-commerce services, including SaaS platforms that facilitate online transactions.
- **US: ADA Title III** — courts increasingly apply to websites and mobile apps. The DOJ has endorsed WCAG 2.1 AA as the benchmark. Private plaintiffs file thousands of lawsuits annually; food service and ordering platforms are frequent targets.
- **UK: Equality Act 2010** — "reasonable adjustments" obligation interpreted to include digital accessibility. WCAG 2.1 AA is the de facto standard.
- **Canada: Accessible Canada Act + provincial laws** — Ontario's AODA requires WCAG 2.0 AA for large organizations and is expanding.
- **Australia: Disability Discrimination Act 1992 (DDA)** — the Maguire v. SOCOG (2000) case established that websites are covered.

## Key Requirements

Focus areas for a <!-- FILL: your application description -->: forms, menus, buttons, notifications, images, interactive dashboards.

### Perceivable

1. **Alt text on all meaningful images** — menu item photos, restaurant logos, category icons. Decorative images get `alt=""`.
2. **Color contrast** — 4.5:1 minimum for normal text, 3:1 for large text (18px+ bold or 24px+ regular). Applies to all themes generated by the dynamic theme system.
3. **Text resizable to 200%** without loss of content or functionality. No horizontal scrolling at 320px viewport width (reflow criterion 1.4.10).
4. **No information conveyed by color alone** — order status indicators (pending/yellow, preparing/blue, ready/green, cancelled/red) must include text labels or icons alongside color.
5. **Captions and text alternatives for non-text content** — if video or audio is added to landing pages or onboarding, captions are required.

### Operable

6. **All functionality keyboard-accessible** — tab navigation through menus, enter/space to submit orders, arrow keys for quantity selectors, escape to close modals.
7. **No keyboard traps** — every modal, drawer, and dropdown must allow escape via keyboard. Focus must return to the triggering element on close.
8. **Focus indicators visible** — never suppress outline with `outline: none` without providing an equivalent visible indicator. Custom focus rings must meet 3:1 contrast.
9. **Skip navigation links** — "Skip to main content" link on all pages, especially the admin dashboard with its sidebar navigation.
10. **Sufficient time for form completion** — session timeouts on order forms and checkout must warn users and allow extension.
11. **Target size: minimum 24x24 CSS pixels** — WCAG 2.2 addition (Success Criterion 2.5.8). Applies to all buttons, links, and interactive controls. Particularly relevant for mobile ordering.
12. **Consistent focus order** — tab order follows visual layout. No focus jumping caused by dynamic content insertion.

### Understandable

13. **Form labels associated with inputs** — every input has a visible `<label>` with `htmlFor` or is wrapped in a `<label>`. Placeholder text is not a substitute for labels.
14. **Error messages identify the field and suggest correction** — "Please enter a valid phone number" not "Invalid input." Errors must be programmatically associated with the field via `aria-describedby` or `aria-errormessage`.
15. **Consistent navigation across pages** — navigation patterns remain in the same relative position. Admin layout and customer layout each maintain internal consistency.
16. **Language attribute on HTML element** — `<html lang="en">` (or appropriate language). If Arabic/French support is added, `lang` must update per page.
17. **Predictable behavior** — no unexpected context changes on focus or input. Form submission only on explicit action.

### Robust

18. **Valid, parseable HTML** — no duplicate IDs, proper nesting, closed tags.
19. **ARIA roles and properties used correctly** — `role="dialog"` on modals, `role="alert"` on error messages, `aria-expanded` on collapsible sections, `aria-current="page"` on active nav items. Prefer semantic HTML over ARIA where possible.
20. **Status messages programmatically determinable** — order confirmation, cart updates, toast notifications must use `role="status"` or `role="alert"` (or `aria-live`) so screen readers announce them.
21. **Name, Role, Value** — all custom components (quantity buttons, toggle switches, star ratings) must expose accessible name, role, and current value to assistive tech.

## Penalties

- **EAA (EU)** — enforcement delegated to member states. Non-compliant services can be removed from the EU market. Penalties must be "effective, proportionate, and dissuasive."
- **ADA (US)** — private right of action (lawsuits). Settlements typically $5,000-$100,000+ for small businesses. Serial plaintiffs target food service and e-commerce platforms. Attorney fees recoverable by plaintiffs.
- **Equality Act (UK)** — civil claims for discrimination. Compensation typically GBP 1,000-10,000+. EHRC can issue enforcement notices.
- **DDA (Australia)** — complaints to Australian Human Rights Commission. Conciliation; if unresolved, Federal Court. Damages and orders to make adjustments.
- **AODA (Canada/Ontario)** — administrative penalties up to CAD 100,000/day for corporations. Public reporting of non-compliance.
